## Executive Summary

- state.json v1.1.0 contains all fields needed to fully generate TODO.md: title, status, task_type, topic, effort, dependencies, artifacts, description, project_number, project_name
- The existing generate-task-order.sh already handles Kahn wave computation + topic-grouped DFS tree — generate-todo.sh should call it (or absorb its logic) rather than duplicating it
- The recommended approach is a **wrapper script** that: (1) writes YAML frontmatter, (2) writes `# TODO` heading, (3) delegates Task Order to generate-task-order.sh, (4) generates the Tasks section by iterating state.json with jq, (5) writes to an atomic temp file then moves it

- Terminal tasks (completed, abandoned, expanded) are excluded from Task Order (already handled by generate-task-order.sh) but must appear in Tasks section for history

## Context & Scope

The current TODO.md is maintained by a suite of scripts that each manipulate different parts:
- `update-task-status.sh` — updates status markers in both the task entry and Task Order section
- `link-artifact-todo.sh` — inserts artifact links into task entries
- `generate-task-order.sh` — regenerates the entire Task Order section

This ad-hoc approach is brittle because any single task entry can get out of sync with state.json (e.g., a field added to state.json after task creation never appears in the existing TODO.md entry). The goal of task 648 is to create `generate-todo.sh` that produces the **entire** TODO.md from scratch from state.json, making it a pure derived artifact. Downstream tasks 649–651 will then simplify the pipeline so that all status/artifact updates only touch state.json, and TODO.md is always regenerated by calling generate-todo.sh.

### 3. Existing generate-task-order.sh Logic

The script is 882 lines with these major components:

1. **Argument parsing**: `--print` (stdout) or `--update-todo TODO STATE` modes
2. **`build_graph()`**: Reads all non-terminal tasks, builds `task_status[]`, `task_deps[]`, `task_desc[]`, `all_task_nums[]` arrays. Uses `title // description // project_name` for the short description in Task Order entries (truncated to 65 chars).
3. **`build_successors_map()`**: Inverts `task_deps[]` to build `task_successors[]`
4. **`load_topics()`**: Loads `topic` fields and `active_topics` order from state.json
5. **`compute_waves()`**: Kahn's BFS algorithm to assign wave numbers
6. **`compute_connected_components()`**: Union-Find for connected components
7. **`generate_wave_table()`**: Produces the `**Dependency Waves**:` markdown table
8. **`generate_grouped_section()`**: Produces the `**Grouped by Topic**:` section with DFS topic trees
9. **`generate_section()`**: Combines timestamp + goal + wave table + grouped section
10. **`replace_section()`**: Finds `## Task Order` in TODO.md and replaces it

**Key filter**: The script uses `select(.status == "completed" | not) | select(.status == "abandoned" | not) | select(.status == "expanded" | not)` to exclude terminal tasks from Task Order.

**Integration approach for generate-todo.sh**:
- Option A (Call): Call `generate-task-order.sh --print` and capture output for Task Order section — simplest, avoids duplicating logic
- Option B (Absorb): Copy all logic inline — eliminates dependency, self-contained file but duplicates ~300 lines
- **Recommendation**: Option A (call with `--print`) — cleaner separation of concerns; generate-task-order.sh already has the complex wave/topic logic tested in production. The caller captures stdout and writes it into the full TODO.md output.

### 5. Proposed Script Interface and Architecture

```
generate-todo.sh [OPTIONS]

Options:
  --todo FILE    Path to TODO.md to write (default: specs/TODO.md)
  --state FILE   Path to state.json to read (default: specs/state.json)
  --dry-run      Print generated TODO.md to stdout instead of writing to file
  --log FILE     Path to log file (default: .claude/logs/generate-todo.log)
  --no-log       Disable logging
```

**Architecture (single-pass generation)**:

```bash
#!/usr/bin/env bash
set -euo pipefail

# 1. Parse arguments, validate files exist

# 2. Open log file, write start entry
log "generate-todo" "START" "state=$STATE_FILE todo=$TODO_FILE"

# 3. Generate YAML frontmatter from state.json
next_num=$(jq -r '.next_project_number' "$STATE_FILE")
printf -- "---\nnext_project_number: %s\n---\n" "$next_num"

# 4. Write fixed header
printf "\n# TODO\n\n"

# 5. Generate Task Order section by calling generate-task-order.sh --print
"$SCRIPT_DIR/generate-task-order.sh" --print

# 6. Write Tasks section header
printf "\n## Tasks\n\n"

# 7. Iterate tasks in descending order (higher number = newer = top)
# Use jq to extract sorted list, iterate with bash
jq -r '[.active_projects[]] | sort_by(-.project_number) | .[].project_number' "$STATE_FILE" |
while read -r task_num; do
  generate_task_entry "$task_num"
  printf "\n---\n\n"
done

# 8. Write to temp file, atomic move
mv "$TMP_FILE" "$TODO_FILE"

# 9. Log success
log "generate-todo" "OK" "wrote $TODO_FILE"
```

**`generate_task_entry()` implementation**:

```bash
generate_task_entry() {
  local num="$1"
  local task_json
  task_json=$(jq -r --argjson n "$num" '.active_projects[] | select(.project_number == $n)' "$STATE_FILE")
  
  local title status effort task_type topic description
  title=$(echo "$task_json" | jq -r '.title')
  status=$(echo "$task_json" | jq -r '.status' | format_status)
  effort=$(echo "$task_json" | jq -r '.effort // ""')
  task_type=$(echo "$task_json" | jq -r '.task_type // "general"')
  topic=$(echo "$task_json" | jq -r '.topic // ""')
  description=$(echo "$task_json" | jq -r '.description // ""')
  
  # Heading
  printf "### %s. %s\n" "$num" "$title"
  
  # Metadata fields
  [[ -n "$effort" ]] && printf -- "- **Effort**: %s\n" "$effort"
  printf -- "- **Status**: [%s]\n" "$status"
  printf -- "- **Task Type**: %s\n" "$task_type"
  [[ -n "$topic" ]] && printf -- "- **Topic**: %s\n" "$topic"
  
  # Dependencies
  local deps_str
  deps_str=$(echo "$task_json" | jq -r '.dependencies // [] | if length == 0 then "None" else map("Task " + tostring) | join(", ") end')
  printf -- "- **Dependencies**: %s\n" "$deps_str"
  
  # Artifacts (grouped by type: research, plan, summary/implementation)
  render_artifacts "$task_json"
  
  # Description
  if [[ -n "$description" ]]; then
    printf "\n**Description**: %s\n" "$description"
  fi
}
```

**Artifact rendering logic**:

Artifacts array in state.json: `[{"type": "research", "path": "...", "summary": "..."}, ...]`

Grouping by `type` field:
- `research` → `**Research**`
- `plan` → `**Plan**`
- `summary`, `implementation` → `**Summary**`
- `report` → `**Research**` (legacy type used in a few tasks)

For each field type present:
- Count artifacts of that type
- If 1: `- **Research**: [path-without-specs-prefix]`
- If 2+: multi-line list format

The path stored in state.json is `specs/NNN_slug/reports/...` — strip `specs/` prefix for the TODO.md link (per existing convention in link-artifact-todo.sh).

### 6. Edge Cases

**Terminal tasks in Tasks section**:
- Tasks with `status = completed | abandoned | expanded` appear in the Tasks section for history
- They are excluded from Task Order (already handled by generate-task-order.sh which filters them out in `get_active_tasks()`)
- In the Tasks section, they appear at their natural position in the descending sort
- They should still show their artifact links and description

**Missing fields**:
- `title` absent: fall back to `project_name` (replace underscores with spaces, title-case) — task 647 should have ensured all tasks have titles, but fallback is prudent
- `description` absent: omit the `**Description**:` block entirely
- `effort` absent: omit the `- **Effort**:` line
- `topic` absent: omit the `- **Topic**:` line
- `dependencies` absent or `[]`: render as `None`
- `artifacts` absent or `[]`: skip all artifact links

**Tasks with no description in state.json**:
- Some older tasks (78, 87, 639) had description migrated from TODO.md as part of task 647
- If description field exists (even if empty string), handle gracefully

**Artifact type mapping gaps**:
- `report` type (used in task 650): map to `**Research**`  
- Unknown types: render as `**{Capitalized type}**:` with path

**Long descriptions with embedded newlines**:
- `jq -r '.description'` preserves newlines verbatim
- TODO.md renders markdown, so embedded newlines in description will create new paragraphs
- The current TODO.md shows descriptions as single blocks after `**Description**:` — no changes needed, just emit the description as-is

**Ordering of task entries in Tasks section**:
- Current TODO.md lists highest task number first (most recently created = top)
- This is the standard viewing order
- Use `sort_by(-.project_number)` in jq

**Atomic write**:
- Write to temp file in same directory as TODO.md (for atomic `mv`)
- Use `mktemp` for temp file name
- Clean up temp file on EXIT trap

## Decisions

1. **Call generate-task-order.sh rather than absorb its logic** — Task Order generation is complex (Kahn waves, union-find, DFS tree), well-tested, and used independently by other commands. Duplication creates maintenance burden. generate-todo.sh calls `generate-task-order.sh --print` and captures its stdout.

2. **Descending task order in Tasks section** — Match existing TODO.md ordering (highest number = newest = top of list). Implemented via `jq '[.active_projects[]] | sort_by(-.project_number)'`.

3. **Strip `specs/` prefix from artifact paths in TODO.md links** — Consistent with existing link-artifact-todo.sh convention. The path `specs/648_create_generate_todo_script/reports/01_generate-todo-research.md` becomes `[648_create_generate_todo_script/reports/01_generate-todo-research.md]` in the link.

4. **Atomic write via mktemp + mv** — Prevents partial writes from leaving TODO.md in a corrupted state. Use `mktemp -p "$(dirname "$TODO_FILE")"` to ensure same filesystem for atomic mv.

5. **Terminal tasks included in Tasks section** — Completed/abandoned/expanded tasks remain visible in the Tasks section as history. They are already excluded from Task Order by generate-task-order.sh.

6. **Log to `.claude/logs/generate-todo.log`** — Matches existing log location pattern. Lightweight append-only log for post-validation review.

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| generate-task-order.sh fails (e.g., no active tasks) | Medium | Check exit code; if it exits 0 with no output (INFO message to stderr), write empty Task Order section. If it exits non-zero, log error and abort. |
| Large state.json (many tasks) causes slow jq calls | Low | Process once in main jq call; avoid per-task jq invocations where possible by loading all tasks into a bash array in one pass |
| Description contains characters that break printf | Medium | Use `printf '%s\n'` not `printf "$description"` — never use user data as printf format string |
| Atomic mv across filesystems fails | Low | Create temp file in same directory as TODO.md (`mktemp -p "$(dirname "$TODO_FILE")"`) |
| Concurrent calls both writing TODO.md | Medium | Both will succeed via atomic mv; last writer wins. This is acceptable since TODO.md is derived — same state.json input always produces same output. |
| Missing `title` field on legacy tasks | Low | Fallback: `project_name | gsub("_"; " ") | ascii_downcase | ltrimstr(" ") | ...` — task 647 enriched all tasks, but defensive fallback is prudent |
