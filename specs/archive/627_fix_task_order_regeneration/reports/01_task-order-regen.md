## Executive Summary

- **Issue 1 (shift 3 bug)**: `generate-task-order.sh` line 53 uses `shift 3` inside the `--update-todo` case of its `while [[ $# -gt 0 ]]` argument parser. Under `set -euo pipefail`, if the script is invoked with fewer than 3 arguments in that position (e.g., `--update-todo` with only 0 or 1 file paths instead of 2), `shift 3` returns non-zero and the script exits immediately via `set -e`. All current callers pass the correct 3 args, but the parser is fragile and a safe `shift` idiom is needed.
- **Issue 2 (active_topics not updated)**: `/meta` (meta-builder-agent), `/fix-it` (skill-fix-it), `/spawn` (skill-spawn), and `/errors` auto-infer or skip topic fields when creating tasks, but **none of them add new topic values to the top-level `active_topics` array in state.json**. `/task` documents the intent ("append to state.json active_topics array before writing task entry") but the jq template in step 6 omits the actual `active_topics` update, leaving it to LLM interpretation.
- **Issue 3 (non-blocking call verification)**: All five task-creating commands include a non-blocking `generate-task-order.sh --update-todo` call after task creation. The calls pass the correct arguments and the script writes to TODO.md via `mv` (not stdout), so the mechanism is sound. However, the calls use **relative paths** (`specs/TODO.md`, `specs/state.json`) which depend on CWD being the project root; if CWD is wrong, the script silently fails (stderr suppressed via `2>/dev/null`). There is also a minor inconsistency: `task.md` create mode uses a `$gen_script` variable, while sync mode and all other commands use `bash` prefix.

## Context & Scope

The Task Order section in `specs/TODO.md` is auto-generated by `.claude/scripts/generate-task-order.sh`. It groups active tasks by topic, computes dependency waves via Kahn's algorithm, and renders a DFS tree per topic. It should be regenerated after every task-creation event. This task addresses three bugs in that pipeline.

## Findings

### 1. generate-task-order.sh Analysis

**File**: `.claude/scripts/generate-task-order.sh`

#### 1a. The shift 3 Bug (Line 53)

```bash
# Lines 43-65
while [[ $# -gt 0 ]]; do
  case "$1" in
    --print)
      MODE="print"
      shift
      ;;
    --update-todo)
      MODE="update"
      TODO_ARG="${2:-}"
      STATE_ARG="${3:-}"
      shift 3          # <-- LINE 53: BUG
      ;;
    --goal)
      GOAL_OVERRIDE="${2:-}"
      shift 2
      ;;
    *)
      echo "Unknown argument: $1" >&2
      ...
      exit 1
      ;;
  esac
done
```

**Behavior under `set -euo pipefail`**: `shift N` returns exit code 1 when `$# < N`. Under `set -e`, this immediately terminates the script. 

**When the bug fires**: If the script is called with `--update-todo` and fewer than 2 file arguments (e.g., `generate-task-order.sh --update-todo` or `generate-task-order.sh --update-todo /path/to/TODO.md`), then `$# == 1` or `$# == 2` at the point of `shift 3`, which fails.

**Current callers**: All current callers pass the correct 3-token sequence (`--update-todo $TODO_FILE $STATE_FILE`), so the bug does not fire in practice today. However, the parser is brittle and would fail silently (due to the `2>/dev/null` wrappers in callers) if any caller were to omit one file path argument. The `--goal` case has the same latent bug: `shift 2` would fail if `--goal` is the last argument with no value.

**Note**: `TODO_ARG="${2:-}"` and `STATE_ARG="${3:-}"` use the `:-` form which handles unset/empty without error, so the variable assignment itself does not fail. Only the subsequent `shift 3` fails.

#### 1b. Topic Loading

The script reads topics correctly from two sources:
- Per-task `topic` field: `jq -r '.active_projects[] | ... | "\(.project_number)|\(.topic // "")"'`
- Canonical order: `jq -r '.active_topics // [] | .[]'` (top-level `active_topics` array)

Tasks with topics not in `active_topics_order` are collected in a secondary pass (lines 371-382) and appended after canonical topics. The `### Uncategorized` section handles tasks with no topic. So the script itself handles missing `active_topics` entries gracefully — it just loses canonical ordering.

#### 1c. How `--update-todo` Writes to TODO.md

The `replace_section()` function (lines 760-809):
1. Finds the line numbers of `## Task Order` (start) and the next `## ` heading (end)
2. Writes lines before the section to `mktemp` file
3. Appends the new section content to the temp file
4. If a section end was found, appends lines from there onward
5. Calls `mv "$tmp_file" "$TODO_FILE"` — atomic replacement

This is correct. Writing happens via `mv`, not via stdout, so redirect of stdout/stderr does not affect the write. The function returns 1 (non-zero) if the `## Task Order` heading is not found, causing the script to exit non-zero.

#### 1d. Other Fragility: Missing Task Order Section

If TODO.md has no `## Task Order` section, `replace_section()` emits a warning to stderr and returns 1. Since callers use `2>/dev/null || echo "Note: ..."`, this failure is silently swallowed. The TODO.md in this repo has `## Task Order` at line 7, so this is not currently a problem.

### 3. update-task-status.sh Analysis

**Phase 3** (`update_todo_task_order` function) handles Task Order updates on status transitions:

- **Mode B (terminal: COMPLETED, ABANDONED, EXPANDED)**: Calls `generate-task-order.sh --update-todo "$TODO_FILE" "$STATE_FILE"` (absolute paths derived from `$SCRIPT_DIR`). Passes correct args.
- **Mode A (non-terminal)**: In-place sed replacement of `[OLD_STATUS]` with `[NEW_STATUS]` on the matched tree line. Falls back to Mode B if the task is not found in the Task Order tree.

The script does NOT itself update `active_topics`. It does NOT handle new task creation (only status updates for existing tasks).

**No issues found in update-task-status.sh itself.** The paths used are absolute (derived from `$SCRIPT_DIR`), making it robust to CWD issues.

### 4. active_topics Semantics

From `task-order-format.md`:
- `active_topics` is a **canonical ordering list** — it controls the display order of topic sections in the Task Order output.
- Topics found in per-task `topic` fields but NOT in `active_topics` are still displayed, just appended in encounter order after the canonical sections.
- The `generate-task-order.sh` script handles this gracefully: "Collect any topics in tasks that aren't in `active_topics_order`" (lines 371-382).

**Conclusion**: `active_topics` is **not a whitelist**; it is an ordering hint. Missing it does not prevent topics from appearing. However, it causes sub-optimal ordering (newly added topics appear at the end rather than in project-defined canonical order). The fix is to ensure that when any command assigns a topic value not already in `active_topics`, it appends that value to the `active_topics` array.

## Decisions

1. **Shift 3 fix**: Replace `shift 3` with a safe pattern that shifts by `min(3, $#)` or restructures the argument consumption so that `shift` is only called for the number of args actually available.

2. **active_topics updates**: Add `active_topics` maintenance to each task-creating command. The best approach is a shared pattern (jq one-liner) that conditionally appends the new topic if not already present. All five commands need this.

3. **Non-blocking call verification**: The calls are mechanically correct (write via `mv`, not stdout). The main risk is relative paths and CWD assumptions. Document this. The fix is to ensure the calls are made from a known CWD or use absolute paths. Since `update-task-status.sh` already uses `$SCRIPT_DIR`-relative absolute paths as the model, the same approach can be recommended for command-level callers (though they are templates, not shell scripts).

## Proposed Fixes

### Fix 1: shift 3 Bug in generate-task-order.sh

**File**: `.claude/scripts/generate-task-order.sh`
**Location**: Lines 49-54

**Current code**:
```bash
    --update-todo)
      MODE="update"
      TODO_ARG="${2:-}"
      STATE_ARG="${3:-}"
      shift 3
      ;;
```

**Proposed fix** (option A — safe shift):
```bash
    --update-todo)
      MODE="update"
      TODO_ARG="${2:-}"
      STATE_ARG="${3:-}"
      # Safe shift: only shift as many positions as available
      local available=$#
      shift $(( available < 3 ? available : 3 ))
      ;;
```

**Proposed fix** (option B — positional consumption):
```bash
    --update-todo)
      MODE="update"
      shift  # consume --update-todo
      TODO_ARG="${1:-}"
      [[ $# -gt 0 ]] && shift
      STATE_ARG="${1:-}"
      [[ $# -gt 0 ]] && shift
      ;;
```

Apply the same pattern to the `--goal` case (lines 55-58):
```bash
    --goal)
      GOAL_OVERRIDE="${2:-}"
      # safe: shift 2 of up to $# remaining
      local available=$#
      shift $(( available < 2 ? available : 2 ))
      ;;
```

**Note**: Option B (sequential shift) is cleaner and avoids the arithmetic. It is the recommended approach.

### Fix 2: active_topics Maintenance in Task-Creating Commands

**Shared jq pattern** to insert after each task's `topic` is determined:

```bash
# If topic is non-empty and not already in active_topics, append it
if [[ -n "$topic" ]]; then
  already_present=$(jq -r --arg t "$topic" \
    'if (.active_topics // [] | index($t)) != null then "yes" else "no" end' \
    specs/state.json)
  if [[ "$already_present" == "no" ]]; then
    jq --arg t "$topic" \
      '.active_topics = ((.active_topics // []) + [$t])' \
      specs/state.json > specs/tmp/state.json && mv specs/tmp/state.json specs/state.json
  fi
fi
```

**Files to update**:

| Command | Location | Action |
|---------|----------|--------|
| `task.md` | Step 6, before the main jq update | Add active_topics append jq call |
| `meta-builder-agent.md` | Stage 6, CreateTasks, before/after each task jq | Add active_topics append per unique topic |
| `skill-fix-it/SKILL.md` | Step 9.1, after topic inference | Add active_topics append per unique topic |
| `skill-spawn/SKILL.md` | Stage 11, if topic is added | Add topic field + active_topics maintenance |
| `errors.md` | Step 4, task creation block | Add topic inference + active_topics append |

**For efficiency with multiple tasks** (meta, fix-it, spawn, errors): Collect all unique new topics first, then do a single jq update to append all at once:

```bash
# Collect unique new topics not already in active_topics
new_topics_json=$(jq -n --argjson arr '["topic1","topic2"]' \
  'reduce $arr[] as $t (
    inputs | .active_topics // [];
    if (index($t)) != null then . else . + [$t] end
  )' specs/state.json)
# Note: use correct jq pattern per jq-escaping-workarounds.md
```

### Fix 3: Non-Blocking Call Robustness

**Problem**: All command-level callers use relative paths (`specs/TODO.md`, `specs/state.json`) which depend on CWD = project root. This is generally correct for Claude Code invocations but worth documenting.

**Recommended approach**: Add a comment in each command noting the CWD dependency, and consider using the `bash` prefix with explicit path detection:

```bash
# Recommended pattern (consistent across all commands):
_script_path=".claude/scripts/generate-task-order.sh"
if [ -f "$_script_path" ]; then
  bash "$_script_path" --update-todo specs/TODO.md specs/state.json \
    2>/dev/null || echo "Note: Failed to regenerate Task Order (non-fatal)" >&2
fi
```

**Inconsistency to fix in task.md**: The create mode (Part C) uses a `$gen_script` variable without `bash` prefix, while the sync mode (step 6) uses direct path. Standardize on the `bash` prefix + `-f` check pattern used by all other commands.
