## Findings

### 1. Current Behavior of generate-task-order.sh

The script works as documented. It:

1. Reads `active_projects` from state.json, filtering out entries with status `completed`, `abandoned`, or `expanded`.
2. Extracts descriptions using `(.description // .project_name) | .[0:65]` — truncated to 65 chars.
3. Builds a dependency graph of only active (non-terminal) tasks.
4. Cleans dependencies to only include edges pointing to other active tasks (terminal deps are treated as satisfied).
5. Runs Kahn's algorithm to compute wave assignments.
6. Renders topic-grouped sections if `topic` fields exist in state.json; falls back to `### Uncategorized` otherwise.
7. Replaces the `## Task Order` section in TODO.md by detecting the start (`^## Task Order$`) and end (next `^## ` heading).

**Verified behavior**: Running `generate-task-order.sh --update-todo` on BimodalLogic with correct paths produces a correct output — task #232 does NOT appear in the generated tree, because it's status `completed` and filtered out.

### 2. Issue 1: Task #232 Still Appears in Task Order (Root Cause)

**Root cause**: Task #232 is in `active_projects` with `"status": "completed"` — it has been marked complete but NOT yet archived by `/todo`. The BimodalLogic Task Order section shown in TODO.md is the OLD, hand-curated section that was never regenerated after #232 was completed.

**Script behavior is correct**: The generate script filters `select(.status == "completed" | not)` so #232 is correctly excluded when the script runs.

**Actual problem**: The `update-task-status.sh` script, when called with `postflight:implement` (which maps to `COMPLETED`), triggers Mode B (full regeneration via `generate-task-order.sh --update-todo`). This regeneration should have removed #232 at that time.

**Why it didn't happen**: The BimodalLogic TODO.md's Task Order section uses custom Track/Tier/Phase headings (not the auto-generated format). The `replace_section()` function in `generate-task-order.sh` finds `## Task Order` and the next `## ` heading correctly, but entirely replaces the hand-curated content with auto-generated content. This probably was not triggered (or was triggered and the user manually reverted it, or the BimodalLogic project uses its own separate copy of the script that points to a different root).

**Key finding**: Confirmed that `update-task-status.sh` does call `generate-task-order.sh --update-todo` for COMPLETED transitions (Mode B). The issue in BimodalLogic is a stale TODO.md state — the section was never regenerated after task #232 was completed, likely because the task was completed before the generate-task-order.sh integration was added, or the hand-curated content predates auto-generation.

### 3. Issue 2: Artifact Paths Instead of Task Descriptions (Root Cause)

**Root cause**: The `build_graph()` function uses this jq expression for descriptions:
```bash
"\(.project_number)|\((.description // .project_name) | ltrimstr(" ") | .[0:65])"
```

When a task has no `description` field, it falls back to `project_name`. The `project_name` field is the task slug (e.g., `copyright_headers_universe_polymorphism_line_limits`, `research_lean4_tactics_infrastructure`). These slugs look like artifact paths or coded names rather than readable descriptions.

**Example observed**:
- Task 180: `description: null`, `project_name: "copyright_headers_universe_polymorphism_line_limits"` → displays as `copyright_headers_universe_polymorphism_line_limits`
- Task 179: `description: null`, `project_name: "research_lean4_tactics_infrastructure"` → displays as `research_lean4_tactics_infrastructure`

**This is NOT an artifact path being substituted**; it is the project_name slug. Tasks created with `/task "description"` should have a description field; older tasks may lack it.

The report description says "artifact paths instead of task descriptions" — this likely refers to seeing slugs that resemble paths or opaque strings rather than actual artifact file paths being inserted. The root cause is that many BimodalLogic tasks have `description: null` in state.json and only have a `project_name` slug.

**Fix**: The script could truncate slugs better (replace underscores with spaces), or the data could be improved by populating `description` fields. A more ergonomic fix would be to un-slugify the project_name fallback: replace `_` with spaces and capitalize first letter.

### 4. Issue 3: BimodalLogic Hand-Curated vs Auto-Generated Format

The BimodalLogic TODO.md's Task Order section uses a custom, manually maintained format:

```
### Track A — Completeness (sorry-free `bx_completeness`)

202 [PLANNED] — Reynolds model surgery (v17): 11-piece, 6-phase plan. ~700 LOC, 16 hours.
  - **Plans**: [specs/202.../plans/...]
...
```

Key characteristics:
- Uses custom `### Track X — Name` headings with narrative text
- Entries include inline artifact links as sub-bullets
- Has Tier/Phase sub-headings as plain text lines within topic sections
- Contains `*Tier 1 — Foundations (independent of each other):*` lines
- Task entries include rich descriptions (not just the 65-char truncation)

**Incompatibility**: `generate-task-order.sh` would replace all this with auto-generated content using topic-grouping from state.json `topic` fields and the standard tree format. The BimodalLogic project's Task Order section is intentionally richer than what the script generates.

**Critical finding**: The script's `replace_section()` function searches for `## Task Order` and the next `## ` line to bound the replacement. The BimodalLogic section ends where `## Tasks` begins — this boundary detection is correct. However, the internal `### Track X` headings are also `### ` headings, and the format spec notes that `### ` headings do NOT trigger the section-end boundary detector (only `^## ` does). So the boundary detection is correct. The problem is that the auto-generated content does not match the hand-curated format.

**Concrete observation**: When `generate-task-order.sh --update-todo` is run against BimodalLogic, it completely replaces the hand-curated Track/Tier/Phase structure with a Kahn-wave-based topic-group tree. Task #228 is shown as `228 [RESEARCHED] — Fix all stale metadata and documentation across data/. (1) Update` rather than the hand-crafted entry. The BimodalLogic project's Task Order section is essentially hand-curated documentation that the auto-generated format cannot replicate.

**Resolution needed**: Either (a) accept auto-generation and populate `topic` fields + `active_topics` order in state.json to approximate the Track structure, or (b) add a `--preserve-headings` mode that only updates task status markers and prunes completed tasks while preserving custom headings and artifact links.

### 5. How link-artifact-todo.sh Interacts with Task Order Section

**Finding**: link-artifact-todo.sh does NOT incorrectly match task numbers in the Task Order section.

The script searches for task entries using:
```bash
heading_line=$(safe_grep -n "^### ${task_number}\." "$TODO_FILE" | head -1 | cut -d: -f1)
```

This pattern `^### {N}.` matches `### 232. Labeled branch infrastructure for world/time-indexed tableau` — a task entry heading in `## Tasks`, NOT the Task Order tree entry `232 [COMPLETED] — ...`.

The Task Order tree entries have format `{N} [STATUS] — {desc}` (at root level) or `  └─ {N} [STATUS] — {desc}` (indented). Neither format matches `^### {N}.`.

**Confirmed**: link-artifact-todo.sh is safe with respect to the Task Order section. It only finds and modifies lines within the specific task's entry block under `## Tasks`.

### 6. Trigger Points for Task Order Regeneration

Current triggers for `generate-task-order.sh --update-todo`:

| Trigger | Condition | Mode |
|---------|-----------|------|
| `update-task-status.sh` | Terminal transition (COMPLETED/ABANDONED/EXPANDED) | Mode B: Full regeneration |
| `update-task-status.sh` | Task not found in Task Order tree (fallback) | Mode B: Full regeneration |
| `skill-todo` Stage 10.5 | After task archival | Full regeneration |
| `skill-todo` Stage 10.5 | After vault operation | Full regeneration (second pass) |
| `skill-fix-it` | After creating tasks | Full regeneration |
| `skill-spawn` | After spawning new tasks | Full regeneration |
| `update-task-status.sh` | Non-terminal transitions | Mode A: In-place status update only |

**NOT triggered by**:
- `/revise` postflight — `update-task-status.sh postflight plan` is non-terminal (→ PLANNED), which uses Mode A only
- Any research/plan/implement postflight — these are non-terminal transitions (Mode A)
- Task creation via `/task` command

**Gap**: The `/revise` postflight calls `update-task-status.sh postflight plan` which sets status to PLANNED (non-terminal). Mode A only updates the status marker in-place in the existing tree. If the task was not already in the Task Order (e.g., newly created and not yet regenerated), Mode A falls back to full regeneration. But if the Task Order is stale in other ways (e.g., the task entry has wrong description), Mode A will not fix it.

## Decisions

1. Issue 1 (completed tasks still in Task Order): The script correctly handles this when called. The BimodalLogic problem is a stale TODO.md that predates or was reset after script integration. Fix is to run `generate-task-order.sh --update-todo` against BimodalLogic, which has been verified to correctly exclude #232.

2. Issue 2 (slug descriptions): A partial fix is to un-slugify the project_name fallback. A proper fix requires populating `description` fields for tasks that lack them.

3. Issue 3 (BimodalLogic custom headings): The auto-generation script cannot preserve hand-curated content. BimodalLogic should either migrate to state.json-driven topic structure, or a `--preserve-headings` mode needs to be added that only updates status markers and removes terminal tasks without reformatting headings.

4. link-artifact-todo.sh is safe; no changes needed.

5. `/revise` does not trigger full regeneration — this is probably acceptable since revise only changes plan status (non-terminal). Full regeneration on every revise would overwrite custom BimodalLogic sections.

## What Changes Are Needed

### Fix 1: Un-slugify project_name fallback in generate-task-order.sh (build_graph)

In `build_graph()`, the description extraction jq:
```bash
"\(.project_number)|\((.description // .project_name) | ltrimstr(" ") | .[0:65])"
```

Should un-slugify the project_name fallback. A bash post-processing step after loading `task_desc` can replace underscores with spaces:
```bash
desc="${desc//_/ }"
```

This makes `copyright_headers_universe_polymorphism_line_limits` display as `copyright headers universe polymorphism line limits`, which is more readable.

### Fix 2: Preserve custom Task Order sections (optional --preserve-headings mode)

For projects like BimodalLogic that maintain hand-curated Task Order sections, add a `--preserve-headings` flag to `generate-task-order.sh` that:
1. Reads the existing Task Order section
2. Extracts all `### ` headings and preserves their surrounding text
3. Only updates: (a) task status markers `[OLD] → [NEW]`, (b) removes lines for completed/abandoned tasks, (c) updates the timestamp line
4. Does NOT replace the section wholesale

This is a significant addition. The simpler alternative is to document that BimodalLogic should migrate its Task Order to be state.json-driven.

### Fix 3: Add Task Order regeneration trigger in /revise postflight (optional)

`skill-reviser` Stage 7 calls `update-task-status.sh postflight plan`. After that, optionally call `generate-task-order.sh --update-todo` to regenerate the Task Order. This ensures description updates from plan revisions are reflected.

However, this would destroy BimodalLogic's custom section on every `/revise`. This should only be done after Fix 2 is in place.

### Fix 4: Populate description fields for tasks lacking them (data quality)

For tasks with `description: null` in state.json, populate the `description` field so the tree displays readable text. This is a data quality fix for existing tasks in BimodalLogic.
