## Executive Summary

- The tree direction reversal requires changes in three locations: `generate-task-order.sh`, `task-order-format.md` (primary copy), and the extension core copy.
- The primary code change is in `_print_topic_node()` and `print_tree_node()`: instead of iterating `task_deps[$task_num]` (prerequisites), they must iterate a new `task_successors` map (dependents).
- A global `task_successors` associative array must be built during `build_graph()` by inverting `task_deps`.
- Root selection in `generate_grouped_section()` needs to change: roots are tasks with **no active deps** (wave 1 tasks within a topic), not tasks that no other task depends on.
- `update-task-status.sh` Mode A regex `^\s*(└─ )?${task_number} \[` is direction-agnostic — it works for both the old and new tree layout without modification.
- The two `task-order-format.md` files are out of sync: the extension core copy is the older pre-May-2026 manual-curation format, while the primary copy is the current auto-generated wave+tree format. Only the primary copy needs updating for this task.

## Context & Scope

The Task Order section in `specs/TODO.md` is auto-generated by `generate-task-order.sh`. It displays active tasks grouped by topic with a DFS tree. The current tree shows each task at depth 0 followed by its prerequisites indented below — meaning a user looking at depth-0 tasks sees tasks they cannot yet work on (they are blocked by the indented children). The desired behavior inverts this: depth-0 tasks are those with **no prerequisites** (wave 1 / unblocked tasks), and their **successors** (tasks that depend on them) are indented below. This way, a user working top-to-bottom naturally works on unblocked tasks first, and as they complete them, the indented successors become promoted to workable.

## Findings

### 1. Current Data Structures and Code Flow

**`build_graph()` (lines 122-177)**
- Builds `task_deps[task_num]` = space-separated string of active dependency IDs.
- Does NOT build a reverse (successors) map — this must be added.
- The `compute_waves()` function (lines 216-276) already builds a local `successors` associative array inside itself. This local variable is thrown away after `compute_waves()` returns and is not available to the tree-printing functions.

**`_print_topic_node()` (lines 420-468) — Primary tree renderer**
```bash

  ...
  for dep in "${dep_array[@]}"; do

  done

```
- Must change to iterate `task_successors[$task_num]` instead of `task_deps[$task_num]`.
- The "cross-topic" annotation logic (lines 438-448) uses `_globally_visited` to detect tasks already printed in another topic section. This logic continues to work correctly in the new direction — when a successor belongs to a different topic, it was already printed there.

**`print_tree_node()` (lines 581-627) — Legacy tree renderer (generate_dependency_tree)**
```bash
# Current: iterates task's dependencies (prerequisites)

  ...
  for dep in "${dep_array[@]}"; do
    print_tree_node "$dep" $((depth + 1))
  done

```
- Same change as above: iterate `task_successors[$task_num]`.

**`generate_dependency_tree()` (lines 629-662)**
- Current root selection: tasks with `task_deps[$tn]` empty (no active deps).
- This is CORRECT for the new behavior — roots are unblocked tasks (no active deps = wave 1). No change needed here.
- The fallback loop (lines 656-660) also continues to work.

**`generate_grouped_section()` (lines 335-416) — Root selection within topics**

This is the critical difference. Currently the function iterates `topic_tasks` in numeric order and calls `_print_topic_node` for each unvisited task. Because the DFS goes *downward into deps*, a task with deps gets printed first (it's the root), and its prerequisites are rendered as children. In the new direction, roots within a topic are tasks with no active deps (within that topic, or globally). The iteration logic does NOT explicitly separate roots from non-roots — it relies on the DFS to avoid re-printing visited tasks.

In the **new direction**:
- When iterating `topic_tasks` in numeric order, a task with no active deps will be printed as a root (depth 0) with its successors indented below.
- A task that IS a successor of another task in the same topic will be visited via DFS from its prerequisite. When the outer loop reaches it, `_topic_section_visited` will already contain it, so it is skipped.
- **This means the existing outer loop iteration logic works correctly WITHOUT change** — the DFS direction change in `_print_topic_node` automatically makes wave-1 tasks appear as roots.

However, there is a subtlety: tasks whose prerequisites are ALL in other topics will appear as roots within their own topic section, since they have no same-topic parents. This is the desired behavior — they are displayed at depth 0 in their topic section with a note that their prerequisite is elsewhere.

### 2. Successors Map Construction

The `compute_waves()` function already builds a `successors` map (local to that function). For the tree rendering, a **global** `task_successors` array is needed. It should be built in `build_graph()` or immediately after as a second pass:

```bash
# New global declaration (alongside task_deps):
declare -A task_successors  # task_num -> space-separated IDs of tasks that depend on it

# In build_graph(), after task_deps is populated — add second pass:
build_successors_map() {
  for tn in "${all_task_nums[@]}"; do
    local deps="${task_deps[$tn]:-}"
    if [[ -n "$deps" ]]; then
      read -ra dep_array <<< "$deps"
      for dep in "${dep_array[@]}"; do
        [[ -z "$dep" ]] && continue
        task_successors["$dep"]="${task_successors[$dep]:-} ${tn}"
      done
    fi
  done
  # Trim leading spaces
  for tn in "${all_task_nums[@]}"; do
    task_successors["$tn"]="${task_successors[$tn]# }"
  done
}
```

This can be called right after `build_graph` in the main section, or incorporated into `build_graph` itself as a second pass.

### 3. Root Selection in `generate_grouped_section()`

The existing iteration logic:
```bash
for tn in "${topic_tasks[@]}"; do
  if [[ -z "${_topic_section_visited[$tn]+x}" ]]; then
    _print_topic_node "$tn" 0
  fi
done
```

In the new direction, this correctly produces:
- Wave-1 tasks (no active deps) will be printed first at depth 0.
- Their successors within the same topic will be printed by DFS from the wave-1 task.
- When the outer loop reaches a task already visited by DFS, it skips it.

**No change is needed** to the outer loop iteration logic. The DFS direction change in `_print_topic_node` propagates the change automatically.

One exception: if a task's ALL prerequisites are completed/terminal, it has no active deps and behaves as a wave-1 task — it will be printed as a root. This is also correct behavior.

### 4. `update-task-status.sh` Mode A Analysis

- In the new tree: task N might appear as a root or as an indented successor. The regex finds it either way.

**No changes needed to `update-task-status.sh`.**

### 6. Edge Cases Analysis

**Cross-topic dependencies (new direction)**:
- If task 599 (workflow-refactor) depends on 598 (workflow-refactor), in the new tree 598 is the root and 599 is an indented child. Both are in the same topic — no cross-topic annotation needed.
- If task 602 (wezterm-notifications) depends on 601 (wezterm-notifications), same topic — 601 is root, 602 is child.
- Cross-topic: if task B (topic-B) depends on task A (topic-A), in the new tree A is printed first in topic-A section. When topic-B section renders, B is a root (its dep A is in another topic), but when printing B's successors, any successors that ARE in topic-A would appear as cross-topic children. The `_globally_visited` check prevents double-printing.

**Diamond dependencies (new direction)**:
- If tasks C and D both depend on A, and E depends on both C and D:
  - A is the root. C and D are successors of A (both indented below A).
  - E depends on C and D. When printing C's subtree, E is printed as a child of C.
  - When printing D's subtree, E is already in `_topic_section_visited`, so it gets `(see above)`.
  - This correctly renders the diamond.

**The `(see above)` annotation** — In the old direction, a dep printed under multiple parents got `(see above)`. In the new direction, a successor printed under multiple parents gets `(see above)`. The semantics are the same; only the relationship direction changed. No code change needed for this logic.

**Tasks with no successors** (leaf nodes in new tree):
- These are the "deepest" work-in-progress tasks. They appear as leaves indented below their prerequisites. They are simply printed with no further recursion. This is correct behavior.

**Tasks with no deps within their topic but deps in other topics**:
- In the new direction, these are still roots within their topic (depth 0), since they have no same-topic predecessors to place them under. When a task in their successor tree belongs to another topic, it gets the `(see above)` annotation. No special handling needed.

## Decisions

1. **Successors map placement**: Build `task_successors` as a separate function `build_successors_map()` called immediately after `build_graph()` in the main section. This keeps `build_graph()` focused on building the primary data structures and avoids duplicating the inversion logic that already exists in `compute_waves()`.

2. **Root selection — no change to outer loop**: The existing `generate_grouped_section()` outer loop logic already works for the new direction. DFS from wave-1 tasks will naturally visit successors; the outer loop skips already-visited tasks. No explicit root-filtering logic is needed.

3. **`generate_dependency_tree()` — update to match**: Even though this function is preserved for debugging, it should be updated consistently for the new direction (root detection already correct; change the DFS to use `task_successors`).

4. **Extension core copy**: The `.claude/extensions/core/context/formats/task-order-format.md` is an old pre-automation format that is no longer the active format. The correct action is to replace it with the current primary copy (post-update) to keep the two files in sync, as the task description requires.

5. **Sorting of successors**: The current code sorts `dep_array` numerically for consistent output. The same numeric sort should be applied to the successors array to maintain deterministic output.
